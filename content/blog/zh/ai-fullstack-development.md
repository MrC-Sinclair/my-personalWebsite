## 引言：当 AI 成为开发流水线的一部分

> "以前开发一个模块，我需要花半天写代码、半天改 bug。现在有了 AI，半天就能从需求走到可演示版本，剩下时间在打磨细节和审查质量。"

这是我引入 AI 辅助开发工作流后的真实感受。根据实际项目数据统计，**新需求开发效率提升了 45%**。但这个数字不是靠"遇到问题就问 AI"这种零散方式实现的，而是通过**构建一套完整的 AI 驱动开发流程**，让 AI 从项目启动的第一天就深度参与每个环节。

AI 全流程开发，简单来说就是：**将 AI 工具系统地嵌入到软件开发的每个阶段——需求分析、架构设计、代码生成、测试、部署、运维——而不是只把它当作一个代码补全插件或 bug 搜索工具。**

这篇文章将以一个真实项目为蓝本，手把手演示完整的 AI 全流程开发怎么做。

## 项目背景：我们这次要做什么

为了演示完整流程，我们以一个典型的**企业内部管理系统模块**为例——**客户管理 CRM 的线索跟进模块**（这也是我简历中实战过的场景之一）。

**项目速览：**

| 项目 | 内容 |
| --- | --- |
| 模块 | CRM 线索管理（Lead Management） |
| 核心功能 | 线索列表、线索录入、跟进记录、转化统计 |
| 技术栈 | Vue3 + TypeScript + ECharts + Element Plus |
| 目标周期 | 传统开发预计 5 天 → AI 全流程预计 2.5 天 |
| 涉及 AI 工具 | Cursor（IDE）、Claude（架构设计）、GitHub Copilot（编码辅助）、CodeBuddy（项目管理） |

现在，让我们一起走完这个项目的 AI 全流程。

## 阶段一：需求分析 —— 让 AI 帮你理清边界

传统做法中，需求分析阶段我们拿到产品 PRD，然后开始画脑图、列功能清单、列接口。这个阶段往往因为需求描述不清晰导致后期返工。

**AI 全流程的做法：**

### 1.1 用 AI 做需求澄清

拿到原始需求后，先把产品描述发给 AI，让它帮你列出潜在的问题和模糊点：

```markdown
Prompt 示例：
"我正计划开发一个 CRM 线索管理模块，核心功能包括线索录入、线索列表、跟进记录、转化统计。
请帮我：
1. 列出该模块可能遗漏的子功能
2. 指出需求描述中的模糊点（例如：线索的状态流转规则是什么？）
3. 给出推荐的功能优先级排序（P0/P1/P2）"
```

当 AI 返回如下问题时，你会在需求阶段就发现关键遗漏点：

- 线索状态流转规则未明确（新线索→跟进中→已转化/已流失）
- 线索查重逻辑缺失（同一个客户可能被多次录入）
- 跟进记录是否支持附件上传？
- 转化统计的时间粒度是什么（日/周/月）？

### 1.2 生成功能清单和接口草案

需求澄清后，让 AI 根据澄清后的需求生成功能清单和初步的 API 接口设计：

```markdown
Prompt 示例：
"请根据以下需求生成：
1. 前端功能清单（含路由设计、组件划分）
2. 对应的 RESTful API 接口列表（含请求/响应结构）
3. 数据模型定义（TypeScript interface）

需求：
- 线索列表页：支持分页、状态筛选、关键字搜索
- 线索录入页：表单填写、查重校验、提交
- 跟进记录：以时间线形式展示，可新增
- 转化统计：柱状图展示每日/每周转化率"
```

这个 Prompt 会生成一份完整的结构文档，你只需要审查和微调即可进入下一阶段。

### 阶段一的关键技巧

- **把 AI 当产品经理用**：让 AI 扮演"挑剔的产品经理"，主动发现需求漏洞
- **一次生成，多轮审查**：不要期望一次 Prompt 就得到完美结果，至少要经过两轮 AI 审查
- **产出物要文档化**：把 AI 生成的接口文档和数据模型存下来，这些会成为后续阶段的上下文

## 阶段二：架构设计 —— AI 作为架构评审师

很多人以为架构设计必须自己从头到尾想好，其实**让 AI 先生成一个方案然后你再评审修正，效率远高于从零思考**。

### 2.1 生成架构方案

```markdown
Prompt 示例：
"你是资深前端架构师。我正在开发一个 CRM 线索管理模块，技术栈 Vue3 + TypeScript + Element Plus + ECharts。
以下是功能清单和 API 接口：[粘贴阶段一输出]

请为我设计：
1. 目录结构方案（推荐 pages、components、composables、types、utils 的划分）
2. 组件树结构（哪些组件是可复用的，哪些是页面独有）
3. 状态管理方案（Pinia store 划分）
4. 数据流图（组件间数据流转）
5. 需要考虑的非功能性设计（权限控制、错误处理、加载态等）"
```

AI 生成的方案可能不完美，但能给你一个**95% 正确的起点**。你只需要：
- 审查合理性（架构师经验仍然不可或缺）
- 根据项目特殊需求做调整
- 把自己思考的关键约束补充进去

### 2.2 设计评审 Dialog

针对 AI 方案中你认为有风险的地方，可以开启"评审模式"：

```markdown
"这个方案中，你建议把线索列表的筛选状态放在 URL query 中，但我们的场景需要实时保存用户的筛选偏好。
请重新评估：是放在 localStorage + URL 双重存储更好，还是只放 localStorage 通过路由守卫还原？分别分析优缺点。"
```

这种交互式的架构评审，可以帮你更快地发现方案盲区。

### 阶段二的关键技巧

- **定义好上下文再提问**：先给 AI 项目的技术栈、约束、已有的架构风格
- **多用"请分析优缺点"**：让 AI 列出多个方案的 trade-off，而不是只给一个"最优解"
- **架构评审至少三轮**：自己的方案 → AI 方案 → 综合最优

## 阶段三：编码实现 —— 核心效率来源

这是 AI 发挥最大价值的阶段。但很多人犯的错误是：**一次性给太多上下文，或者上下文太少**。

### 3.1 模板型页面：交给 AI 全权负责

表单、表格、列表这类模板化页面，AI 的完成度最高。

```markdown
Prompt 示例：
"基于 Element Plus，创建一个线索录入表单组件 CreateLeadForm.vue。

要求：
- TypeScript 类型安全，类型定义从 @/types/lead 导入
- 表单字段：姓名、手机号（正则校验）、来源渠道（下拉选择）、备注（多行文本）
- 级联联动：选择'线上渠道'时额外显示'来源平台'字段
- 手机号输入时自动调用查重接口（防抖 500ms），重复时红色警告
- 提交时 loading 状态，成功/失败通过 ElMessage 提示
- 表单验证使用 Element Plus 的 rules
- 使用 Composition API + script setup 语法
- style 使用 scoped，颜色使用 Design Token 变量"
```

注意这个 Prompt 的构成：
1. **技术栈约束**：Element Plus、TypeScript、Composition API
2. **业务逻辑细节**：级联联动、查重防抖
3. **组件规范**：Design Token、scoped style
4. **交互要求**：loading、消息提示

**越具体的约束，AI 生成的代码质量越高。**

### 3.2 复杂逻辑：AI 写骨架，你填血肉

对于复杂的数据处理或交互逻辑，让 AI 先生成骨架，你再填充关键部分。

```markdown
Prompt 示例：
"请为我生成线索转化统计图表组件 ConversionChart.vue 的骨架代码：

功能需求：
1. 使用 ECharts 绘制柱状图，显示近 7 天/近 30 天的转化率
2. 支持日/周/月视图切换（选项卡切换）
3. 数据通过 props 传入，格式为 { date: string, rate: number }[]
4. 窗口 resize 时自动重绘
5. 导出组件类型定义

请先定义 props 的 TypeScript 接口，再生成组件代码。
关键的业务计算逻辑（如转化率公式）用 TODO 标记，我后续补充。"
```

这样 AI 帮你把 80% 的样板代码写完——事件监听、组件结构、响应式适配——你只需要填充核心业务逻辑。

### 3.3 接口对接：AI 的强项

对接后端 API 是前端开发中最枯燥但 AI 最擅长的工作。给定接口文档，AI 可以一气呵成生成完整的请求函数、错误处理和类型定义：

```markdown
Prompt 示例：
"根据以下 API 定义，生成前端请求函数和对应的 Vue 组件集成代码：

POST /api/leads
  请求体：{ name: string, phone: string, source: string, remark?: string }
  响应：{ code: number, data: { id: number }, message: string }

GET /api/leads?page=1&pageSize=20&status=string&keyword=string
  响应：{ code: number, data: { list: Lead[], total: number } }

要求：
1. 使用 axios 实例（从 @/utils/request 导入）
2. 所有请求函数统一放在 @/api/lead.ts 中导出
3. 统一的错误处理（ElMessage.error）
4. 完整的 TypeScript 类型定义
5. 包含 loading 状态管理"
```

### 3.4 AI 辅助调试

遇到 bug 时，与其自己一行一行排查，不如把错误信息 + 关键代码段发给 AI：

```markdown
"我在线索列表的分页功能中遇到了问题：
- 切换页码后，筛选条件丢失
- URL query 和组件状态不同步

代码片段如下：[粘贴关键代码]
控制台报错：[粘贴报错信息]

请帮我分析原因并给出修复方案。"
```

**经验之谈**：80% 的 bug AI 都能在第一个答案中定位到根因。如果 AI 连续两次答错，那可能是一个需要你亲自断点调试的"深度 bug"。

### 阶段三的关键技巧

- **按模块分 session**：不要在同一个对话中处理多个不相关的模块，每个功能模块用新的对话
- **先做抽象层，再做具体实现**：先让 AI 定义类型、工具函数、API 封装，再让它使用这些抽象层生成组件
- **增量式修改优于重写**：告诉 AI "在上次的基础上增加 X 功能"，而不是每次重新描述完整需求
- **审查每个 diff**：不要盲目接受 AI 的修改，用 IDE 的 diff 视图逐行审查

## 阶段四：测试覆盖 —— AI 最被低估的能力

单元测试是 AI 的"舒适区"。写好测试用例是质量保障的关键环节。

### 4.1 生成单元测试

```markdown
Prompt 示例：
"请为以下 utility 函数生成 vitest 单元测试：

函数：formatLeadStatus(status: string): string
作用：将线索状态的值转换为中文显示文本
映射关系：new→新线索, contacting→跟进中, converted→已转化, lost→已流失

要求：
1. 覆盖所有状态值（含边界值）
2. 覆盖异常情况（空字符串、未定义、未知状态）
3. 使用 describe/it/expect 组织
4. 显式写出测试名称"
```

AI 生成的测试通常能覆盖 90% 以上的分支。审查时重点关注**遗漏的边界情况**。

### 4.2 生成组件测试

```markdown
Prompt 示例：
"请为 CreateLeadForm.vue 组件生成 vitest 测试：

需要测试的场景：
1. 表单必填项为空时提交，应显示验证错误
2. 手机号格式错误时提交，应显示手机号格式错误提示
3. 手机号输入时，查重接口应被调用（防抖 500ms）
4. 查重接口返回重复时，应显示重复警告
5. 提交成功时，应触发 'submit-success' 事件
6. 提交失败时，应显示错误消息

请使用 @vue/test-utils + vitest，模拟依赖的数据/接口。"
```

### 阶段四的关键技巧

- **让 AI 先写测试再写实现（TDD）**：这是 AI 全流程开发中最推荐的做法，测试定义了行为边界
- **覆盖率目标 80%+**：AI 生成测试 + 人工补充边界情况，覆盖率很容易达到
- **测试也要 Review**：AI 可能生成"通过但不检验"的虚假测试，检查 assert 是否真的有意义

## 阶段五：集成、部署与性能优化

### 5.1 路由与权限集成

用 AI 生成路由配置和权限校验高阶组件：

```markdown
Prompt 示例：
"基于 Vue Router，为线索管理模块配置路由：

路由表：
- /leads → 线索列表（需要 'leads:view' 权限）
- /leads/create → 线索录入（需要 'leads:create' 权限）
- /leads/:id → 线索详情（需要 'leads:view' 权限）

请：
1. 使用动态路由导入（import.meta.glob）
2. 添加路由元信息（meta.roles、meta.title、meta.icon）
3. 集成路由守卫进行权限校验
4. 添加路由过渡动画"
```

### 5.2 性能分析

在开发完成后，让 AI 帮你做性能审查：

```markdown
"请审查以下线索列表组件的代码，识别可能的性能问题：
1. 是否有不必要的重渲染？
2. 列表数据量大时，虚拟滚动是否必要？
3. 是否有内存泄漏隐患？
4. 是否有可优化的请求策略（预加载、缓存）？

代码：[粘贴代码]"
```

### 5.3 构建与部署配置

```markdown
"请为这个 Vue3 + TypeScript 项目生成优化后的 vite 构建配置：

要求：
1. 代码分包策略（vendor chunk、按路由懒加载）
2. gzip 压缩插件配置
3. 去掉 console.log（生产环境）
4. 图片资源压缩
5. 打包分析（rollup-plugin-visualizer）

当前项目结构：[粘贴目录结构]"
```

## 全流程复盘：AI 做了什么，你做了什么

| 阶段 | AI 负责 | 你需要做 |
| --- | --- | --- |
| 需求分析 | 发现需求漏洞、生成功能清单 | 确认需求、优先级决策 |
| 架构设计 | 生成目录结构、组件树、数据流图 | 架构评审、关键决策 |
| 编码实现 | 生成 80% 的代码（模板页面、API 对接、测试） | 核心业务逻辑、代码审查 |
| 测试 | 生成 90% 的测试用例 | 补充边界情况、审查覆盖率 |
| 部署 | 生成构建配置、CI/CD 配置 | 验证配置、处理兼容问题 |

回到开头的效率数据：**效率提升 45%** 不是 AI 替代了你 45% 的工作，而是 AI 承担了**大量重复性、模板化的工作**，让你能将精力集中在更需要人类判断力的高价值活动上。

## 避坑指南：AI 全流程开发的常见陷阱

### 陷阱一：跳过架构评审

**表现**：让 AI 直接生成代码，不看整体方案，结果页面虽能工作但风格不一致、组件不可复用。

**解决**：哪怕只花 10 分钟，也要让 AI 先输出架构方案，审查后再开工。

### 陷阱二：Prompt 过于模糊

**表现**："帮我写个线索列表"→ AI 输出一个基础表格，然后反复修改 6-7 轮。

**解决**：第一次 Prompt 就给出**技术栈、组件库、数据字段、交互方式、设计规范**五要素。

### 陷阱三：AI 生成代码直接上线

**表现**：AI 代码在 DEMO 环境跑通了 → 上生产 → 出现安全漏洞或内存泄漏。

**解决**：建立 AI 代码审查清单（边界条件、错误处理、类型安全、安全漏洞、性能），逐项检查。

### 陷阱四：所有模块一视同仁

**表现**：用同样的 Prompt 策略处理表单页面和复杂图表逻辑。

**解决**：**模板型代码交给 AI，核心逻辑谨慎对待。** 对于涉及大量业务判断的代码，AI 负责骨架，核心逻辑人工填充。

## 结语：构建属于你的 AI 全流程

AI 全流程开发不是一套固定的模板，而是一种**思维方式**：在每个开发环节中问自己"这件事 AI 能不能帮我做"，然后找到让 AI 介入的最佳方式。

回到我的实际经验——从最初拿 AI 写正则表达式，到后来用 AI 生成完整模块，再到现在的全流程 AI 协作——这个过程不是一蹴而就的。关键是一步步建立信任：**先用 AI 做小事建立信心，再逐步扩大 AI 的参与范围。**

最终你会发现，AI 全流程开发的核心竞争力不是代码生成，而是：

1. **需求拆解能力**：把复杂需求拆成 AI 能处理的小块
2. **审查能力**：快速判断 AI 输出质量的眼光
3. **上下文管理能力**：给 AI 恰到好处的信息量
4. **工程决策能力**：知道哪些事必须自己做，哪些事可以交给 AI

> AI 不会取代开发者，但会用 AI 的开发者会取代不会用 AI 的开发者。而把 AI 系统地嵌入全流程的开发者，会成为效率最高、产出质量最稳定的那个。
